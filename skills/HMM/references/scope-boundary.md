# scope-boundary

## 目标

把 [agentic-nested-state-machine.opml](agentic-nested-state-machine.opml) 从“大而全的系统蓝图”压缩为一个可独立工作的 skill MVP 边界。

本文件只解决一件事：

- 什么内容进入 skill 主流程
- 什么内容先降级为扩展规则
- 什么内容明确移出 skill
- 什么内容转入 `references/`

## 四层拆分

### 第一层：核心 skill 流程

定义：不依赖复杂 UI、不依赖多 agent 协同、不依赖外部记忆治理，也能独立完成一次“问题结构化 -> 方案规划 -> 评估回路”的最小流程。

只保留 6 个主状态：

1. `问题接收`
2. `语义澄清`
3. `需求约束建模`
4. `分支计划生成`
5. `评估校验`
6. `循环或结束`

这些状态是 skill 主体，未来必须写成严格的输入输出契约。

### 第二层：agent 编排扩展

定义：不是 skill 能否成立的前提，但能提升复杂任务处理效率的编排能力。

MVP 只保留最少规则，不保留复杂实现设想。

当前仅保留以下最小编排规则：

- 仅当子问题之间无控制依赖时才允许分支并行
- 分支内部默认串行，一次只推进一个明确步骤
- 向用户提问时优先选择式、启发式，而不是开放式追问
- 若本轮评估判定问题定义不充分，则回到问题定义阶段
- 同一轮尽量不要求模型解决超过 3 步的任务

以下编排能力暂不进入 MVP 主流程，只作为后续扩展候选：

- 共享文件驱动的异步协作
- 历史记忆编辑与替换
- 子 agent / 并 agent 类型细分
- 多分支投票、层次分析法聚合
- 批注工具与环境观察式协同

### 第三层：UI / 产品设想

定义：面向界面交互、可视化编辑、视图组织、文本操作体验的内容。

这一层全部移出 skill，不写入 `SKILL.md` 主体，不作为触发条件，不作为执行步骤。

### 第四层：理论参考

定义：帮助模型理解分析方法，但不是每次触发 skill 都必须完整加载的理论、术语、方法库和分析框架。

这一层应进入 `references/`，按需加载，而不是塞进 skill 主流程。

## OPML 内容归层

### 进入第一层：核心 skill 流程

来自 `agentor` 主树，保留为 MVP 主流程的内容：

- 优化提示词
- 提示词语义澄清校准
- 提示词意图需求校准
- 首次搜索
- 真需求-真约束方法论
- `P = {S_0, S_g, O, C}` 状态空间框架中的最小问题定义部分
- 问题陈述、定义问题
- 研究具体计划
- 执行计划中的“模板变量输入”最小能力
- 评价 agent 中的基础校验能力
- 解决流程中的“无效则回到分析阶段”

说明：

- 旧 `metaPrompt` 的线性 `步骤 1 到步骤 8` 已经被新的 chapter-based `metaPrompt v2` 取代。
- 从当前边界看，最适合直接进入 MVP 主流程的是：
  - 第 `4` 章 `任务入口与术语澄清`
  - 第 `5` 章 `问题建模`
  - 第 `6` 章 `状态、约束与理想解`
  - 以及压缩壳层中的基础六状态
- 第 `7` 章到第 `11` 章已经是 maintained source 的一等方法层，不应再被当成“旧稿附录”。
- 但在当前 skill MVP 壳层中，它们仍应按需展开，而不是默认每轮全量激活。

### 默认核心与按需展开的边界

为了避免把 maintained source 的“存在”误读成“必须每轮全开”，当前仓库采用以下分层：

#### 默认核心

默认每轮都应稳定存在：

- 第 `4` 章 `任务入口与术语澄清`
- 第 `5` 章 `问题建模`
- 第 `6` 章 `状态、约束与理想解`
- 第 `10` 章的最小评估意识
- 第 `12` 章的主链与回退规则

#### 一等层但按需展开

以下层已经是 maintained source 的核心组成，但在 MVP 壳层里仍可按需展开：

- 第 `7` 章 `资源与系统层级`
- 第 `8` 章 `检索与证据补全`
- 第 `9` 章 `解生成`
- 第 `10` 章 `评估、筛选与落地`
- 第 `11` 章 `插件沉淀与方法自升级`

判断原则：

- 如果当前任务的主链被这些层阻断，则它们必须显式展开；
- 如果当前任务无需它们，也不能假装它们不存在，只能保持“未激活但已定义”的状态。

### 进入第二层：agent 编排扩展

来自 `ai人间体的社交分工网络` 主树，作为扩展规则保留：

- 并行分支
- 并行
- 分支串行
- 悬置遍历
- 异步在哪些场景发挥作用
- 用户交互节点
- 标准节点内必有的功能中的流程规则部分
- 解决流程中的循环逻辑
- 共创中的分支并行、分支串行、分支合并

说明：

- 这些内容描述的是“如何组织分析过程”，不是“要分析什么”。
- 它们属于编排层，不该与 ANT、SAFC、TRIZ 这类分析层混写。

### 进入第三层：UI / 产品设想

以下内容全部移出 skill：

- 多个方案排序与颜色区分展示
- 方案块抽取、编辑、组合为方案四
- 文本高亮、浮标、长按选择、拖动抖动
- `TextKit2`
- `SwiftUI ScrollView`
- `UICollectionView + CompositionalLayout`
- 多个 Text Viewport 拼接
- 左右切换方案视图
- 底部按钮按方案执行

说明：

- 这些内容是未来产品界面设计，不是 skill 的 procedural knowledge。
- 如果保留在 skill 中，会污染触发条件和主流程边界。

### 进入第四层：理论参考

以下内容转入 `references/`：

- 行动者网络理论 `ANT`
- `EAFC`
- `SAFC`
- `TRIZ`
- `USIT32`
- 流分析
- 多屏幕分析法
- 因果链、三元组、有向无环图、数学逻辑符号格式
- 参数库、效应库、临时库
- 根需求、心理学、社会学、行为经济学效应库
- 五向接近最优方法论
- 多目标贝叶斯优化、帕累托、鲁棒优化、加权法
- IFR、问题拉力、反向目标、否定剪枝法

说明：

- 这些内容对 skill 很重要，但不应默认全量进入上下文。
- 后续应拆成多个 reference 文件，按任务需要加载。

## MVP 主流程边界

### 状态 1：问题接收

目标：

- 接收用户原始问题
- 判断是单步问题还是多步问题
- 收集最小必要背景

保留输入：

- 用户原始描述
- 用户立场
- 已知约束

不做：

- 不直接展开多路径大规模探索
- 不直接调用复杂理论

### 状态 2：语义澄清

目标：

- 术语去歧义
- 重表述问题
- 提取核心名词与可替代术语

保留内容来源：

- `metaPrompt v2` 第 `4` 章
- `opml` 中“优化提示词”“提示词语义澄清校准”

### 状态 3：需求约束建模

目标：

- 明确现状、目标状态、行动者、约束、资源、核心摩擦
- 形成最小问题模型

保留内容来源：

- `metaPrompt v2` 第 `5` 章与第 `6` 章
- `opml` 中“真需求-真约束方法论”
- `P = {S_0, S_g, O, C}` 的最小子集

不做：

- 不在此状态引入完整 SAFC、TRIZ、USIT 求解

### 状态 4：分支计划生成

目标：

- 把问题拆成若干可执行子问题
- 判断哪些可以并行，哪些必须串行
- 形成计划，不做深度执行

保留内容来源：

- 并行 / 串行规则
- 分支并行、分支串行
- 本轮聚焦 todo

### 状态 5：评估校验

目标：

- 检查问题定义是否完整
- 检查分支是否合理
- 输出基础质量判断

保留内容来源：

- 同源一致性校验
- 自打分评估
- 拆解至事实级验证链

限制：

- 自打分只能用于排除明显问题，不能代替真实验证

### 状态 6：循环或结束

目标：

- 若信息不足或结构有误，回退到前面状态
- 若产物完整，则结束本轮

保留内容来源：

- “如果解决无效，则重新回到分析问题阶段”

## 明确移出的内容

以下内容在 MVP 阶段一律不进入主流程：

- 深层 UI 交互与客户端实现
- 多账号 agent 社交分工网络
- 历史对话直接编辑与记忆替换
- 复杂投票与自动权重决策
- 共享文件异步协调机制
- 批注工具驱动协同
- SAFC 求解、USIT32、TRIZ 矛盾求解的完整执行链
- 大规模多方法论穷举探索

## 当前边界下的最小独立能力

删掉 UI 设想和非必要编排后，这个 skill 仍应能独立完成：

1. 把模糊问题重写为结构化问题
2. 给出需求、约束、资源、行动者、摩擦的最小模型
3. 输出一个可执行的分支计划
4. 对计划做基础校验
5. 判断继续迭代还是结束

如果做不到这 5 点，说明当前边界仍然过宽或主流程仍然混入了扩展层内容。

同时，对齐 `metaPrompt v2` 之后，还应额外满足：

6. 当关键信息缺失时，能显式进入检索与证据补全，而不是把“搜索”当成一句口头提醒；
7. 当高频结构稳定出现时，能显式留下插件沉淀信号，而不是把“插件化”永远推迟到外部说明里。
