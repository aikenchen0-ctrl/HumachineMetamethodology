# `metaPrompt` 元方法论主文件 `v2`

## 第 0 章 文档定位与适用边界

### 0.1 目标

定义本方法论是什么、解决什么、不解决什么。

### 0.2 输入

无固定输入。本章是整套方法论的总说明。

### 0.3 核心动作

1. 把方法论定位为上位框架，而不是单一提示词。
2. 明确它同时处理：
   - 问题定义；
   - 系统建模；
   - 约束与理想解；
   - 检索与证据补全；
   - 候选解生成；
   - 评估与落地；
   - 插件沉淀与 `skill` 化。
3. 明确其默认目标不是操控、剥削或提取，而是提高系统理想度、降低有害功能、提升问题解决与任务达成的稳定性。

### 0.4 输出

1. 方法论适用边界；
2. 方法论非适用边界；
3. 方法论总目标。

### 0.5 常见错误

1. 把本方法论理解成单一提示词模板。
2. 把 `USIT` 五大算子误当成整套方法论。
3. 把插件层误当成主文件本体。
4. 把检索工具误当成方法论本身。

### 0.6 衔接

本章输出进入第 `1` 章，冻结共享语言。

## 第 1 章 核心对象与符号系统

### 1.1 目标

统一整套方法论的共享语言，避免后续章节出现概念混写。

### 1.2 输入

1. 原始术语集合；
2. 材料间冲突符号；
3. 需要共享的核心对象。

### 1.3 核心动作

1. 冻结共享符号。
2. 定义每个符号只表示什么。
3. 建立禁止混用清单。

### 1.4 统一符号表

| 符号 | 定义 | 作用层 |
| --- | --- | --- |
| `S0` | 当前状态、现状 | 状态层 |
| `Sg` | 目标状态、现实可达目标 | 状态层 |
| `IFR` | 理想终态、理想解极限 | 理想解层 |
| `O` | 改变系统路径的操作、机制或手段 | 变换层 |
| `E` | 实体、行动者、关键对象 | 建模层 |
| `A` | 属性、参数化特征 | 建模层 |
| `F+` | 正向功能结果 | 建模层 |
| `F-` | 有害功能结果 | 建模层 |
| `Cond` | 功能成立的隐式条件 | 条件层 |
| `C_user` | 用户显式约束 | 约束层 |
| `C_forbid` | 禁区约束 | 约束层 |
| `C_base` | 基底约束、硬边界 | 约束层 |
| `C_comp` | 代偿约束、路径副代价 | 约束层 |
| `R` | 资源集合 | 资源层 |
| `TC` | 跨参数倒挂矛盾 | 矛盾层 |
| `PC` | 参数自互斥矛盾 | 矛盾层 |
| `P` | 领域插件 | 插件层 |
| `Ev` | 证据集合 | 检索与验证层 |

### 1.5 禁止混用清单

1. `Cond` 不等于 `C_user / C_forbid / C_base / C_comp`
2. `IFR` 不等于 `Sg`
3. `E` 不等于 `F`
4. `A` 不等于 `F`
5. `R` 不等于 `O`
6. `P` 不等于 `MCP skill`

### 1.6 输出

1. `symbol_table`
2. `layer_map`
3. `forbidden_mix_list`

### 1.7 常见错误

1. 把条件和约束写成同一个对象。
2. 把理想终态和现实目标混成一层。
3. 把插件和工具混成一层。

### 1.8 衔接

本章输出作为第 `2/5/6/8/9/11` 章的共享语言接口。

## 第 2 章 总骨架

### 2.1 目标

建立整套方法论的上位骨架。

### 2.2 输入

1. 第 `1` 章冻结的符号系统；
2. `Six-Box/CrePS` 骨架认识。

### 2.3 核心动作

1. 固定 `Six-Box Scheme`。
2. 区分 `Real World` 和 `Thinking World`。
3. 明确每一箱在主链中的作用。

### 2.4 六箱定义

| `Box` | 含义 | 世界 |
| --- | --- | --- |
| `Box 1` | 原始任务或原始问题情境 | `Real World` |
| `Box 2` | 被定义清楚的问题 | `Thinking World` |
| `Box 3` | 对当前系统与理想系统的理解 | `Thinking World` |
| `Box 4` | 新系统想法或候选机制 | `Thinking World` |
| `Box 5` | 概念解、组合解、插件解 | `Thinking World` |
| `Box 6` | 现实可执行方案 | `Real World` |

### 2.5 主迁移链

`Box 1 -> Box 2 -> Box 3 -> Box 4 -> Box 5 -> Box 6`

### 2.6 输出

1. 六箱结构定义；
2. 各箱职责；
3. 默认迁移链。

### 2.7 常见错误

1. 把 `USIT` 算子层误当成总骨架。
2. 跳过 `Box 3` 直接从问题跳到方案。
3. 把 `Box 5` 的概念解误写成现实实施方案。

### 2.8 衔接

本章输出进入第 `3` 章工作模式和第 `12` 章通用执行序列。

## 第 3 章 工作模式

### 3.1 目标

让方法论能根据任务类型切换工作模式，而不是一律套同一流程。

### 3.2 输入

1. 当前任务；
2. 当前目标；
3. 当前不确定性。

### 3.3 核心动作

识别主模式，并允许一个次模式并行。

### 3.4 四种模式

| 模式 | 适用场景 |
| --- | --- |
| 问题求解模式 | 有明确目标，需要提出方案 |
| 资料搜索模式 | 关键知识不足，必须补证 |
| 领域插件构建模式 | 某领域会反复出现，需要沉淀插件 |
| 复盘升级模式 | 已有方案或插件表现不稳，需要回收缺陷 |

### 3.5 输出

1. `primary_mode`
2. `secondary_mode`

### 3.6 常见错误

1. 在四种模式之间无边界跳转。
2. 一开始就进入插件模式，忽略主链问题定义。
3. 在信息已足够时仍持续停留在搜索模式。

### 3.7 衔接

本章输出进入第 `4` 章，决定任务入口的具体工作方式。

## 第 8 章 检索与证据补全

### 8.1 目标

把“信息不足”转写为结构化检索任务，并把证据回流到主链。

### 8.2 输入

1. `entity_table`
2. `attribute_table`
3. `function_table`
4. `condition_table`
5. `constraint_table`
6. `contradiction_table`
7. `priority_targets`
8. `plugin_context`，可为空

### 8.3 核心动作

1. 识别信息缺口；
2. 判断是否必须搜索；
3. 定义证据源优先级；
4. 定义 `MCP skill` 触发点；
5. 检索；
6. 回流；
7. 停止检索。

### 8.4 检索触发条件

至少命中以下任一项时，应启动检索：

1. 关键术语定义不清；
2. 关键结构关系未知；
3. 关键参数缺失；
4. 关键约束来源无法确认；
5. 关键方案或验证标准缺失。

### 8.5 检索对象

#### 信息缺口表

| 字段 | 含义 |
| --- | --- |
| `gap_id` | 缺口编号 |
| `gap_type` | 术语/结构/参数/约束来源/现有方案/验证标准 |
| `gap_desc` | 缺口描述 |
| `why_blocking` | 为什么阻碍主链 |

#### 证据源优先级表

| 字段 | 含义 |
| --- | --- |
| `source_id` | 来源编号 |
| `source_type` | 官方文档/标准/论文/案例/数据/代码/网页 |
| `priority` | 优先级 |
| `why_preferred` | 优先原因 |

### 8.6 插件、`MCP skill` 与证据源的分工

1. 插件：
   - 告诉系统去哪里看、先看什么、哪些词最关键。
2. `MCP skill`：
   - 执行查询、读取、提取、分析。
3. 证据源：
   - 被查询的对象。

### 8.7 回流机制

检索结果至少允许回流到：

1. 第 `5` 章；
2. 第 `6` 章；
3. 第 `9` 章；
4. 第 `11` 章。

### 8.8 停止条件

应显式停止搜索，当：

1. 关键缺口已被补齐；
2. 当前已足以恢复主链；
3. 继续搜索不会显著降低不确定性。

### 8.9 输出

1. `gap_table`
2. `search_plan`
3. `source_priority_table`
4. `evidence_writeback`
5. `search_stop_rules`

### 8.10 常见错误

1. 把检索写成泛泛提醒。
2. 搜索结果不回流。
3. 插件、`MCP skill`、证据源不分层。

### 8.11 衔接

本章输出回流第 `5/6/9/11` 章。

### 10.8 衔接

本章输出进入第 `11` 章，并作为 `Box 6` 的现实方案层。

## 第 11 章 插件沉淀与方法自升级

### 11.1 目标

把单次任务中的高频结构、证据源、算子模式和失败模式沉淀为插件和后续 `skill` 接口。

### 11.2 输入

1. `gap_table`
2. `source_priority_table`
3. `candidate_pool`
4. `combination_hints`
5. `solution_paths`
6. `validation_plan`
7. `case_feedback`，可为空

### 11.3 核心动作

1. 判断是否值得插件化；
2. 判断插件级别；
3. 抽取高频术语、结构、约束、证据源和算子模式；
4. 建立插件草案；
5. 建立插件回流；
6. 形成 `skill` 接口线索。

### 11.4 插件分级

1. `P0`
2. `P1`
3. `P2`

默认应先从 `P0 -> P1 -> P2` 逐级沉淀。

### 11.5 插件字段最小集

| 字段 | 含义 |
| --- | --- |
| `plugin_name` | 插件名 |
| `plugin_level` | `P0/P1/P2` |
| `domain_scope` | 领域边界 |
| `task_types` | 适用任务类型 |
| `core_entities` | 核心实体 |
| `core_attributes` | 核心属性 |
| `core_functions` | 核心功能 |
| `core_constraints` | 常见约束 |
| `evidence_sources` | 优先证据源 |
| `operator_map` | 推荐算子映射 |
| `failure_modes` | 失效模式 |
| `upgrade_rule` | 升级条件 |

### 11.6 插件与 `MCP skill` 的分工

1. 插件：
   - 负责领域知识结构；
   - 负责检索上下文；
   - 负责高频结构沉淀。
2. `MCP skill`：
   - 负责执行动作；
   - 负责查询、提取、验证、分析。

### 11.7 回流机制

插件至少应支持四类回流：

1. 词汇回流；
2. 分析回流；
3. 解法回流；
4. 失败回流。

### 11.8 `skill` 外壳

建议后续 `skill` 最少包含：

| 字段 | 含义 |
| --- | --- |
| `skill_name` | 名称 |
| `goal` | 目标 |
| `trigger` | 触发条件 |
| `inputs` | 输入 |
| `outputs` | 输出 |
| `core_steps` | 核心步骤 |
| `tool_policy` | 工具调用策略 |
| `plugin_dependency` | 依赖插件 |
| `failure_modes` | 常见失败模式 |
| `upgrade_rule` | 升级规则 |

### 11.9 输出

1. `plugin_decisions`
2. `plugin_drafts`
3. `plugin_interfaces`
4. `plugin_feedbacks`
5. `plugin_upgrade_rules`

### 11.10 常见错误

1. 把插件写成第二份主文件。
2. 把 `MCP skill` 当成插件本身。
3. 插件不回流，只做静态附录。

### 11.11 衔接

本章输出进入下一轮任务的第 `4/8/9` 章，以及未来 `skill` 体系。

## 第 12 章 通用执行序列

### 12.1 主链

默认主链为：

1. 第 `4` 章 任务入口与术语澄清
2. 第 `5` 章 问题建模
3. 第 `6` 章 状态、约束与理想解
4. 第 `7` 章 资源与系统层级
5. 第 `8` 章 检索与证据补全
6. 第 `9` 章 解生成
7. 第 `10` 章 评估、筛选与落地
8. 第 `11` 章 插件沉淀与方法自升级

### 12.2 强制回退

出现以下任一情况，必须暂停并回退：

1. 某章输出无法成为下一章输入；
2. 检索结果没有回流；
3. 候选解无法比较；
4. 插件层无法判断是否值得沉淀；
5. 方法整体开始无限扩章。

### 12.3 成功判据

当一轮任务结束时，至少应能回答：

1. 问题是否被结构化；
2. 关键约束和矛盾是否被识别；
3. 候选解是否被系统生成；
4. 现实路径是否被比较和收束；
5. 是否产生了插件或 `skill` 线索。
