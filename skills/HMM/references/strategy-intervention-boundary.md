# strategy-intervention-boundary

## Purpose

This file separates optional strategy and intervention logic from the default neutral problem-structuring core.

It exists because the OPML and `metaPrompt.md` still place ideas such as:

- demand visibility shaping,
- field construction,
- adoption-pressure design,
- north-star / barrier framing,
- and non-rational acceleration mechanisms

close to the core method chain.

These ideas may be useful.
But they should not remain implicitly fused with the default core.

## Core Boundary

Treat the following as optional strategy/intervention material rather than default problem-structuring core:

- manufacturing demand visibility,
- field construction,
- default-path construction,
- adoption pressure and social proof,
- deliberate scarcity or queue design,
- non-rational amplifiers,
- and strategic extraction / lock-in framing.

The default core should remain neutral and descriptive:

- clarify the problem,
- model demand, constraints, actors, and harmful functions,
- branch plans,
- evaluate coherence,
- and serialize stable outputs.

Optional strategy/intervention material should activate only when:

- the user explicitly asks for strategy design,
- the task is about demand creation or visibility shaping,
- or the task explicitly wants field-construction logic rather than neutral analysis.

## Strategy / Intervention Classes

### 1. Demand Visibility Shaping

Examples:

- demand is manufactured into visibility
- latent demand naming
- blocked demand expression

This is not the same as neutral demand discovery.

### 2. Field Construction

Examples:

- constructing a field that pressures action,
- making one path feel like the default path,
- using ranking, certification, or comparison systems to stabilize adoption.

### 3. Adoption Pressure Design

Examples:

- north-star and barrier framing,
- social proof,
- scarcity,
- ranking pressure,
- queue or invitation design.

### 4. Non-Rational Acceleration

Examples:

- authority,
- conformity,
- loss aversion,
- emotional amplification,
- memory fixation.

## Promotion Rule

Do not promote this plane into the default core simply because the source contains powerful strategic language.

Promote it only when:

- the user intentionally wants intervention design,
- the domain explicitly calls for field construction or adoption shaping,
- and the task can tolerate the extra authority and risk boundary this introduces.

Otherwise:

- keep it archived as optional strategy/intervention material,
- and let the neutral core continue to own ordinary problem structuring.

## Current Source-Aligned Decision

The source material around:

- demand visibility,
- field construction,
- default-path creation,
- non-rational amplifiers,
- and north-star / barrier framing

should now be treated as a dedicated optional plane rather than as default core method logic.
